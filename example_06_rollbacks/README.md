# Пример 6: Откаты миграций

## Описание
Демонстрация откатов миграций в различных сценариях. Показывает, как безопасно откатывать изменения схемы базы данных.

## Модель Employee

Модель сотрудника с полями:
- `id`: уникальный идентификатор
- `first_name`: имя
- `last_name`: фамилия
- `email`: электронная почта (уникальная)
- `phone`: телефон (добавлено в миграции 002)
- `department`: отдел (добавлено в миграции 003)
- `position`: должность (добавлено в миграции 003)
- `salary`: зарплата (добавлено в миграции 004)
- `hire_date`: дата найма (добавлено в миграции 004)
- `is_active`: активен ли сотрудник (добавлено в миграции 005)

## Цепочка миграций

```
base -> 001 -> 002 -> 003 -> 004 -> 005 (head)
        ↓      ↓      ↓      ↓      ↓
      table  phone  dept/  salary  active
                    position
```

## Что такое откат миграции?

**Откат (downgrade)** — это процесс отмены изменений, внесённых миграцией. Каждая миграция содержит две функции:

- **upgrade()** — применяет изменения
- **downgrade()** — отменяет изменения

Пример из миграции 002:

```python
def upgrade() -> None:
    op.add_column('employees', sa.Column('phone', sa.String(20), nullable=True))

def downgrade() -> None:
    op.drop_column('employees', 'phone')
```

## Пошаговая инструкция

### Шаг 1: Применение всех миграций

Примените все миграции до последней версии:

```bash
alembic upgrade head
```

Это создаст таблицу `employees` со всеми полями.

### Шаг 2: Проверка текущей версии

```bash
alembic current
```

Вывод:
```
005 (head)
```

### Шаг 3: Просмотр истории

```bash
alembic history --verbose
```

Вы увидите цепочку всех 5 миграций.

### Шаг 4: Откат на одну версию назад

Откатите последнюю миграцию (005):

```bash
alembic downgrade -1
```

Проверьте текущую версию:
```bash
alembic current
```

Теперь вы на версии `004`. Поле `is_active` было удалено из таблицы.

### Шаг 5: Откат на несколько версий

Откатите ещё 2 миграции назад:

```bash
alembic downgrade -2
```

Текущая версия: `002`. Поля `department`, `position`, `salary`, `hire_date` удалены.

### Шаг 6: Откат к конкретной версии

Откатитесь к самой первой миграции (укажите хеш `001`):

```bash
alembic downgrade 001
```

Или сразу к базе (удалит таблицу):

```bash
alembic downgrade base
```

### Шаг 7: Применение миграций вперёд

Вернитесь на несколько версий вперёд:

```bash
alembic upgrade +2
```

Или сразу к последней:

```bash
alembic upgrade head
```

### Шаг 8: Откат к конкретной ревизии по хешу

Если вы хотите откатиться к конкретной миграции:

```bash
alembic downgrade 003
```

Это откатит миграции 005, 004 и остановится на 003.

## Команды для откатов

| Команда | Описание |
|---------|----------|
| `alembic downgrade -1` | Откатить одну миграцию назад |
| `alembic downgrade -N` | Откатить N миграций назад |
| `alembic downgrade <revision>` | Откатиться к конкретной ревизии |
| `alembic downgrade base` | Откатить все миграции (до базы) |
| `alembic upgrade +1` | Применить одну миграцию вперёд |
| `alembic upgrade +N` | Применить N миграций вперёд |
| `alembic upgrade head` | Применить все миграции до последней |
| `alembic current` | Показать текущую версию |
| `alembic history` | Показать историю миграций |

## Важные правила при откатах

### 1. Всегда пишите корректные downgrade()

**ПЛОХО:**
```python
def downgrade() -> None:
    pass  # Ничего не делает!
```

**ХОРОШО:**
```python
def downgrade() -> None:
    op.drop_column('employees', 'phone')
```

### 2. Порядок операций важен

При откате операции выполняются **в обратном порядке**:

```python
def upgrade() -> None:
    op.add_column('employees', sa.Column('department', ...))
    op.add_column('employees', sa.Column('position', ...))

def downgrade() -> None:
    # Обратный порядок!
    op.drop_column('employees', 'position')
    op.drop_column('employees', 'department')
```

### 3. Внешние ключи и индексы

Сначала удаляйте зависимости, потом основные объекты:

```python
def downgrade() -> None:
    # Сначала удалить индекс
    op.drop_index('idx_employee_email')
    # Потом удалить столбец
    op.drop_column('employees', 'email')
```

### 4. Откат не всегда возможен без потери данных

Если миграция удаляет столбец:

```python
def upgrade() -> None:
    op.drop_column('employees', 'old_field')

def downgrade() -> None:
    # Столбец создастся, но данные потеряны!
    op.add_column('employees', sa.Column('old_field', sa.String(100)))
```

**Решение:** Не удаляйте столбцы сразу. Сначала сделайте их nullable и не используйте. Удалите в следующей версии после бэкапа.

### 5. Тестируйте откаты

Всегда проверяйте, что downgrade() работает:

```bash
# Применить миграцию
alembic upgrade +1

# Сразу откатить
alembic downgrade -1

# Применить снова
alembic upgrade +1
```

## Когда использовать откаты?

**✅ Используйте откаты:**
- При обнаружении ошибки в миграции
- Для тестирования изменений схемы
- При переключении между ветками в Git
- Для отладки проблем с БД

**❌ Не используйте откаты в продакшене:**
- Если в таблице есть важные данные (риск потери)
- Если миграция уже давно применена (лучше создать новую миграцию)
- Если не уверены в последствиях (лучше сделать бэкап)

## Безопасные откаты

### Перед откатом в продакшене:

1. **Создайте бэкап базы данных:**
```sql
pg_dump -U postgres -d demo_rollbacks_db > backup.sql
```

2. **Проверьте откат на dev-окружении**

3. **Изучите downgrade() функцию миграции**

4. **Убедитесь, что нет активных пользователей**

5. **Подготовьте план восстановления**

### После отката:

1. Проверьте структуру таблиц
2. Проверьте, что приложение работает
3. Проверьте логи на ошибки
4. Убедитесь, что данные не потеряны

## Визуализация процесса откатов

### Применение миграций (upgrade):
```
base -> 001 -> 002 -> 003 -> 004 -> 005
        ↑      ↑      ↑      ↑      ↑
     upgrade()
```

### Откат миграций (downgrade):
```
base <- 001 <- 002 <- 003 <- 004 <- 005
        ↓      ↓      ↓      ↓      ↓
    downgrade()
```

### Пример: откат с 005 до 002
```
Текущая: 005
Выполнить: alembic downgrade 002

Alembic выполнит:
005.downgrade() -> версия 004
004.downgrade() -> версия 003
003.downgrade() -> версия 002

Результат: версия 002
```

## Типичные ошибки при откатах

### Ошибка 1: Забыли написать downgrade()

```python
def downgrade() -> None:
    pass  # ❌ Ничего не делает!
```

**Последствия:** Откат не работает, схема не изменяется.

### Ошибка 2: Неправильный порядок операций

```python
def downgrade() -> None:
    op.drop_column('employees', 'department')  # ❌ Сначала удалить!
    op.drop_index('idx_department')  # Индекс уже не существует!
```

**Правильно:**
```python
def downgrade() -> None:
    op.drop_index('idx_department')  # ✅ Сначала индекс
    op.drop_column('employees', 'department')  # Потом столбец
```

### Ошибка 3: Потеря данных

```python
def upgrade() -> None:
    op.drop_column('employees', 'old_salary')  # Данные удалены!

def downgrade() -> None:
    op.add_column('employees', sa.Column('old_salary', ...))  # Данные не вернутся!
```

**Решение:** Сначала мигрируйте данные в новый столбец, потом удаляйте старый.

## Заключение

Откаты — мощный инструмент, но требуют осторожности:

- ✅ Всегда пишите корректные downgrade() функции
- ✅ Тестируйте откаты на dev-окружении
- ✅ Делайте бэкапы перед откатами в продакшене
- ✅ Понимайте, что откат может привести к потере данных
- ✅ Используйте откаты для исправления ошибок и тестирования

**Главное правило:** Если не уверены — не откатывайте, а создайте новую миграцию!
