# Практическое занятие №9: Миграция баз данных 🏁

Миграция баз данных необходима при разработке схемы базы данных с течением времени без потери существующих данных. По мере роста и изменения веб-приложений схема базы данных может нуждаться в обновлениях для соответствия новым функциям или требованиям к данным.

Миграции базы данных — это процесс управления изменениями в структуре базы данных, таким образом, чтобы можно было безопасно и последовательно обновлять её, не потеряв данные. Это напоминает систему контроля версий для кода, например, Git. Если что-то пойдет не так, всегда можно откатиться к предыдущей версии.

<div align="center">
  <img alt="Project Demo" src="./mygif/gif-9.1.gif" />
</div>

Важно понимать, что миграция схемы базы данных — это не то же самое, что перенос данных между базами данных. Миграция схемы базы данных затрагивает изменение структуры таблиц, индексов, связей, добавление или удаление колонок и т.д. В отличие от переноса данных, миграция схемы фокусируется на эволюции самой структуры данных, а не на их перемещении между различными системами.

Если вы даже не слышали об этом термине, то можно прочитать вводную часть этой [**статьи**](https://habr.com/ru/articles/121265/)!

Простыми словами, миграции БД — это аналог GitHub для баз данных (хранятся истории изменений, и можно откатиться назад, если что-то сломалось). В данном уроке мы будем рассматривать именно этот процесс, а не переезд базы данных из одного дата-центра в другой или переход с MySQL на MongoDB (если кто так подумал).

Alembic — это полезный инструмент для миграции баз данных, который идеально интегрируется с FastAPI. Он позволяет управлять изменениями схемы базы данных контролируемым и версионным образом. Также стоит отметить, что Alembic был создан автором SQLAlchemy, одной из самых популярных ORM-библиотек для Python. В этом уроке мы будем использовать именно SQLAlchemy, чтобы продемонстрировать работу Alembic.

В зависимости от выбранной вами базы данных или ORM, возможно, вам придется использовать другие инструменты для миграций, такие как `peewee-migrations` для Peewee ORM. В PonyORM миграции появились с версии 0.7.9 и теперь идут "из коробки", что также является удобным вариантом.

## 🎾 SQLAlchemy: введение для начинающих

**SQLAlchemy** — это, пожалуй, самая популярная библиотека для работы с базами данных в Python, реализующая паттерн ORM (Object Relational Mapping). ORM позволяет описывать таблицы базы данных в виде классов Python, что значительно упрощает взаимодействие с данными в приложении. Однако, в этом уроке наша задача только научиться создавать таблицы в нашей базе данных при помощи миграций.

Для установки алхимии пишем:

```bash
pip install sqlalchemy>=2.0
```

### 🍍 Что такое модели SQLAlchemy?

Модель SQLAlchemy — это класс Python, который представляет таблицу в базе данных. Такой класс создаётся путём наследования от декларативного базового класса. Атрибуты модели описывают столбцы таблицы, их типы данных и ограничения. Для определения столбцов используются аннотации типов и функции, позволяющие явно указать характеристики каждого поля, что делает код понятным и удобным для поддержки.

### 🍒 Как создать модель?

Чтобы создать модель `SQLAlchemy`, необходимо:

- Импортировать необходимые классы и функции.
- Создать базовый класс, используя декларативный подход.
- Определить класс-модель, указав имя таблицы через атрибут `__tablename__` и описав столбцы с помощью соответствующих типов и ограничений.

Ниже приведён пример модели, соответствующей таблице `users`:

```python
# myproject/app/db/database.py
from sqlalchemy.types import String
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column

class Base(DeclarativeBase):
    pass

class User(Base):
    __tablename__ = 'users'
    
    id: Mapped[int] = mapped_column(primary_key=True)
    username: Mapped[str] = mapped_column(String(50), nullable=False)
    email: Mapped[str] = mapped_column(String(100), nullable=False)
```

В этом примере:

- Класс `User` представляет таблицу `users`.
- Атрибут `id` — целочисленный столбец, являющийся первичным ключом.
- Атрибуты `username` и `email` — строковые столбцы с заданными ограничениями, что гарантирует обязательное заполнение и соблюдение длины.

> [!WARNING]
> **Важно:** если у вас осталась таблица users с предыдущего урока, то перед началом миграций эту таблицу следует удалить вручную (например через psql: DROP TABLE IF EXISTS users;).

### 🧸 Отличия моделей SQLAlchemy от моделей Pydantic

Важно понимать, что модели SQLAlchemy и модели Pydantic выполняют разные задачи:

- **Модели SQLAlchemy** описывают структуру базы данных, определяя таблицы, столбцы и связи между ними. Они используются для взаимодействия с базой данных посредством ORM.

- **Модели Pydantic** применяются в FastAPI для валидации, сериализации и десериализации данных, поступающих в приложение. Они обеспечивают корректность данных на уровне запросов и ответов, но не влияют на схему базы данных.

Надеемся, что эти знания помогут вам научиться создавать таблицы при помощи миграций.

## 🍓 Начало работы с Alembic

Чтобы начать использовать Alembic для миграции баз данных в вашем приложении FastAPI, выполните следующие действия:

**Шаг 1: Установите Alembic**

Установите библиотеку Alembic, добавив её в зависимости вашего проекта.

```bash
pip install alembic
```

Также вам может понадобиться синхронный драйвер базы данных, чтобы избежать проблем с асинхронной работой Alembic. Для PostgreSQL установите `psycopg2`.

**Шаг 2: Инициализируйте Alembic**

Находясь в корне вашего FastAPI приложения, **обязательно** запустите консольную команду инициализации Alembic, чтобы создать необходимую структуру каталогов Alembic в вашем проекте.

```bash
alembic init alembic
```

**Шаг 3: Настройте Alembic**

После инициализации Alembic **сам** создаст следующую структуру каталогов:

```bash
your_project_directory/
    alembic/
        env.py
        README
        script.py.mako
        versions/
    alembic.ini
```

Настройте Alembic в соответствии с настройками вашей базы данных, включая URL-адрес базы данных, драйвер и другие соответствующие сведения. В частности, вам нужно будет указать модели таблиц (обычно они располагаются в `db/database.py`), где должны быть указаны метаданные из SQLAlchemy. Также настройте файл `env.py` для подключения к базе данных.

**Настройка `alembic.ini`:** Файл `alembic.ini` содержит настройки для подключения к вашей базе данных. В разделе [database] укажите путь к вашей базе данных с помощью переменной `sqlalchemy.url`. Например:

```bash
[alembic]
script_location = alembic

[database]
# Исправьте URL на настоящий для вашей БД
sqlalchemy.url = DATABASE_URL
```

**Рекомендация по безопасности:** Для хранения чувствительных данных, таких как `DATABASE_URL`, рекомендуется использовать файл `.env`. Это позволит не хранить эти данные в коде, а загрузить их в процессе работы приложения, например, с помощью библиотеки `python-dotenv`. В `.env` можно указать такой параметр:

```bash
DATABASE_URL=postgresql://user:password@localhost/dbname
```

Затем в `alembic.ini` можно использовать эту переменную:

```bash
sqlalchemy.url = %(DATABASE_URL)s
```

**Настройка `env.py`:** В файле `env.py` нужно убедиться, что Alembic знает, где искать модели SQLAlchemy. Для этого настройте подключение к базе данных, указав правильный `DATABASE_URL`. Примерно так это может выглядеть:

```python
from logging.config import fileConfig
from sqlalchemy import engine_from_config
from sqlalchemy import pool
from alembic import context
import myproject.app.db.database  # импортируем свои модели

# Чтение конфигурации и установление соединения
config = context.config
fileConfig(config.config_file_name)
target_metadata = myproject.db.database.Base.metadata

def run_migrations_online():
    # Создаем подключение к базе данных
    connectable = engine_from_config(
        config.get_section(config.config_ini_section),
        prefix='sqlalchemy.',
        poolclass=pool.NullPool
    )

    with connectable.connect() as connection:
        context.configure(
            connection=connection,
            target_metadata=target_metadata
        )

        with context.begin_transaction():
            context.run_migrations()

run_migrations_online()
```

Стадию подготовки завершили.

## 🚧 Создание миграций

После настройки Alembic вы можете приступить к созданию миграций, которые представляют собой изменения в схеме вашей базы данных. Миграция — это скрипт на Python, который определяет, как применять изменения (upgrade) и, при необходимости, отменять эти изменения (downgrade).

### 🗂️ Автоматическое создание миграции

Для создания миграции в автоматическом режиме используйте команду:

```bash
alembic revision --autogenerate -m "initial migration"
```

Эта команда создает файл миграции, который можно просмотреть и при необходимости отредактировать. Alembic сравнивает текущие модели с текущим состоянием базы данных и генерирует соответствующие изменения. Однако стоит отметить, что автоматическая генерация не всегда идеальна. Alembic может не распознать некоторые изменения, особенно если они сложные или нестандартные. В таких случаях нужно вручную поправить миграцию или вовсе создать её с нуля.

Пример миграции для создания таблицы `users`:

```python
from alembic import op
import sqlalchemy as sa

# Revision идентифицирует этот миграционный скрипт
revision = '001'
down_revision = None
branch_labels = None
depends_on = None

def upgrade():
    op.create_table(
        'users',
        sa.Column('id', sa.Integer, primary_key=True),
        sa.Column('username', sa.String(50), nullable=False),
        sa.Column('email', sa.String(100), nullable=False),
    )

def downgrade():
    op.drop_table('users')
```

В этом примере, функция `upgrade()` создает таблицу `users` с полями `id`, username и `email`. Функция `downgrade()` содержит код для отката изменений и удаления таблицы с помощью `op.drop_table()`.

### 🎬 Ограничения автогенерации

Важно помнить, что автоматическая генерация миграций с помощью `--autogenerate` имеет свои ограничения. Alembic не всегда правильно определяет изменения, если они не были явно выражены в SQLAlchemy моделях. Например, если вы изменяете тип данных столбца, который уже был создан, Alembic может не обнаружить это изменение. В таких случаях рекомендуется проверять сгенерированный скрипт миграции и при необходимости вручную внести правки.

Если Alembic не видит изменения, выполните следующие шаги:

- Убедитесь, что ваши модели SQLAlchemy синхронизированы с базой данных.
- Проверьте, что миграция не была уже применена, и что схема базы данных актуальна.
- Проверьте файл `env.py`, чтобы убедиться, что он правильно конфигурирует соединение с базой данных и использует актуальные метаданные моделей.

### 🎸 Применение миграции

После создания миграции примените её с помощью команды:

```bash
alembic upgrade head
```

## 🎂 Рабочий процесс Alembic

Типичный рабочий процесс использования Alembic включает в себя следующие этапы:

**Шаг 1: Создайте миграцию**

Сгенерируйте новый сценарий миграции, используя интерфейс командной строки Alembic (CLI). Сценарий фиксирует изменения, которые необходимо внести в схему базы данных.

```bash
alembic revision --autogenerate -m "описание изменений"
```

**Шаг 2: Просмотрите процесс миграции**

Проверьте сгенерированный сценарий миграции, чтобы убедиться, что он точно отражает предполагаемые изменения. Просматривайте и редактируйте сгенерированные миграции, если нужно, чтобы избежать ошибок.

**Шаг 3: Примените миграцию**

Запустите команду Alembic для миграции, чтобы применить изменения к схеме базы данных:

```bash
alembic upgrade head
```

Команда `head` указывает на самую последнюю миграцию. Также можно указать конкретный номер миграции (например, `alembic upgrade 1234abcd5678`), либо указать, на сколько миграций нужно продвинуться, например, `alembic upgrade +2`, чтобы применить две миграции подряд.

**Шаг 4: Проверьте миграцию**

После применения миграции убедитесь, что схема базы данных была обновлена правильно. Для этого можно использовать инструменты базы данных или проверить ее с помощью SQLAlchemy.

## 📬 Управление версиями и ревизии

Alembic поддерживает таблицу версий в базе данных, отслеживая текущую версию миграции. Каждая миграция имеет уникальный идентификатор ревизии. Alembic использует эту информацию для определения порядка, в котором следует применять миграции или отменять их.

**Перескок на конкретную версию**

Если необходимо перескочить на конкретную версию миграции, вы можете использовать команду alembic upgrade, указав номер ревизии. Например:

```bash
alembic upgrade <revision>
```

Где `<revision>` — это уникальный идентификатор миграции, на которую вы хотите перейти. Это позволяет вам переходить к конкретному состоянию базы данных, даже если между текущей и целевой версией есть несколько промежуточных миграций.

### 🗿 Ревизии и зависимости

Каждая миграция в Alembic содержит два важных параметра:

- **down_revision** — указывает на предыдущую миграцию, с которой эта миграция связана. Он определяет, от какой версии миграции эта новая будет "отталкиваться" и позволяет Alembic правильно применить миграции в нужном порядке.

- **depends_on** — позволяет указать, что эта миграция зависит от других миграций, кроме предыдущей. Например, если вы хотите, чтобы эта миграция зависела от двух других, то в `depends_on` можно указать их ревизии. Пример:

```bash
depends_on = ['abc123', 'def456']
```

Этот параметр гарантирует, что миграция не будет применена, пока не будут выполнены все зависимости, указанные в `depends_on`.

Пример миграции с зависимостями:

```python
# Revision идентифицирует этот миграционный скрипт
revision = '002'
down_revision = '001'
depends_on = ['abc123', 'def456']

def upgrade():
    # Здесь ваш код миграции
    pass

def downgrade():
    # Код отката миграции
    pass
```

В этом примере миграция с ревизией '`002`' не может быть применена, пока не будут выполнены миграции с ревизиями '`abc123`' и '`def456`'.

Эти параметры помогают Alembic правильно отслеживать, какие миграции нужно применить или отменить, если были изменены схемы базы данных.

## ⛄️ База данных: наполнение данными (seeding)

В дополнение к миграциям, Alembic позволяет создавать исходные данные для базы данных. Исходные данные — это скрипты, которые заполняют базу данных тестовыми (начальными) данными.

Соответственно, данные = сиды, а заполнение = seeding (засеивание).

Для того чтобы добавить начальные данные в базу, можно использовать команду `op.execute()` или более эффективно — метод `op.bulk_insert()`. С помощью этого метода можно вставить сразу несколько записей в одну операцию.

Пример кода для сидинга через `op.bulk_insert()`:

```python
from alembic import op
import sqlalchemy as sa
from sqlalchemy.sql import table, column
from sqlalchemy import String, Integer

# Ревизия и связи с предыдущими миграциями
revision = 'add_initial_data'
down_revision = 'previous_revision_id'
branch_labels = None
depends_on = None

# Определение структуры таблицы для сидинга
users_table = table(
    'users',
    column('id', Integer),
    column('username', String),
    column('email', String),
    column('password', String)
)

def upgrade():
    # Начальные данные для добавления в таблицу users
    initial_data = [
        {'id': 1, 'username': 'admin', 'email': 'admin@example.com', 'password': 'hashed_password'},
        {'id': 2, 'username': 'user', 'email': 'user@example.com', 'password': 'hashed_password'}
    ]
    # Вставка данных в таблицу
    op.bulk_insert(users_table, initial_data)

def downgrade():
    # Удаление начальных данных в случае отката
    op.execute('DELETE FROM users WHERE username IN ("admin", "user")')
```

В этом примере создается миграция для добавления начальных данных в таблицу `users` с помощью `op.bulk_insert()`. Этот метод позволяет эффективно вставлять несколько строк данных за один раз. В случае отката (`downgrade`) данные удаляются из таблицы с помощью SQL-запроса `DELETE`.

Таким образом, сидинг позволяет вам заполнять базу данными, которые могут понадобиться в процессе разработки или тестирования.

## 🌽 Откаты

Если миграция вызывает проблемы или ее необходимо отменить, Alembic предоставляет механизм отката. Это позволяет вам отменить изменения, внесенные в результате конкретной миграции.

Для отката к предыдущей версии базы данных используйте команду:

```bash
alembic downgrade <revision>
```
                  
Где `<revision>` — это идентификатор миграции, к которой вы хотите откатиться. Например, если вы хотите вернуться к миграции с идентификатором `001`, выполните:

```bash
alembic downgrade 001
```
                  
Также можно откатить изменения на несколько шагов назад, указав относительное количество версий:

```bash
alembic downgrade -2
```
                  
Это откатит вас на две версии назад.

**Когда откатывать миграцию?**

**Откат миграции** — это мощный инструмент, но его следует использовать осторожно, особенно если в базе данных уже есть важные данные, которые могут быть затронуты изменениями. Например, если вы создаете таблицу или добавляете столбцы, откат может вызвать потерю данных, если структура базы данных была изменена. В таких случаях важно убедиться, что все данные, которые могут быть затронуты, безопасно сохранены или перенесены, прежде чем выполнять откат.

Интеграция Alembic с FastAPI позволяет легко управлять схемой базы данных при сохранении данных, однако всегда проверяйте корректность откатов, особенно если это касается сложных операций, таких как удаление или изменение данных.
